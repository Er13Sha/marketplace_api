1. Что это за проект и что он умеет

Это серверная часть (бэкенд) интернет-магазина — а точнее, его ограниченный контекст, отвечающий за каталог товаров. Сейчас Catalog умеет работать не только с товарами, но и с категориями: создавать категории, создавать товары с необязательной категорией, получать товары списком, фильтровать их по категории и менять категорию уже существующего товара.

ВозможностьHTTP-эндпоинтЧто делает
Создать товарPOST /api/catalog/productsСохраняет товар, цену, остаток и необязательную ссылку на категорию.
Получить список товаровGET /api/catalog/productsВозвращает товары; поддерживает q, categoryId, limit, offset.
Получить товар по idGET /api/catalog/products/{id}Возвращает одну карточку товара вместе с категорией, если она задана.
Привязать категорию к товаруPATCH /api/catalog/products/{id}/categoryМеняет категорию существующего товара или снимает её, если передан null.
Создать категориюPOST /api/catalog/categoriesСохраняет категорию с уникальным slug.
Получить список категорийGET /api/catalog/categoriesВозвращает категории для витрины и админ-панели.

Аналогия. Представьте склад с папками. Товар — это карточка позиции, категория — раздел каталога. Создание товара — «оформить карточку», создание категории — «завести раздел», а привязка категории — «переложить карточку в нужный раздел, не меняя сам товар».

Технологии: язык PHP 8.4, фреймворк Symfony 8.1, база данных PostgreSQL (через библиотеку Doctrine), быстрый кэш Redis. Дальше разберём, что значит каждое слово и зачем оно тут.

2. Что такое Symfony (и зачем нужен фреймворк)

Фреймворк — это готовый «каркас» приложения. Без него вам пришлось бы самим писать кучу скучного кода: как принять HTTP-запрос, как разобрать адрес (URL), как достать данные из базы, как вернуть ответ. Фреймворк всё это уже умеет — вы пишете только свою бизнес-логику.

Symfony — один из двух самых популярных PHP-фреймворков. Из того, что он даёт нашему проекту, важны четыре вещи:

Механизм SymfonyЧто делает простыми словами
Роутинг (Routing)Сопоставляет адрес запроса (/api/catalog/products) с конкретным методом в коде, который его обработает.
Контейнер / DI«Сборщик» объектов. Сам создаёт нужные классы и передаёт их друг другу. Подробнее — в разделе 7.
Messenger«Почта» внутри приложения: пересылает сообщения (команды, запросы, события) от отправителя к обработчику.
DoctrineПереводчик между объектами PHP и таблицами базы данных (ORM). Вы работаете с объектом Product, а он сам пишет SQL.
Главная мысльSymfony — это про технику (HTTP, база, очереди). DDD (следующий раздел) — это про то, как организовать бизнес-логику, чтобы она не превратилась в кашу. Эти две вещи дополняют друг друга.

3. Что такое DDD и какую проблему он решает

DDD (Domain-Driven Design, «проектирование на основе предметной области») — это подход к организации кода. Главная идея простая: в центре приложения должны быть бизнес-правила, а не база данных и не фреймворк.

Проблема, которую он решает

В типичном «быстром» приложении бизнес-правила размазаны повсюду: немного в контроллере, немного в SQL-запросе, немного в шаблоне. Пока проект маленький — нормально. Когда он растёт, такой код становится трудно менять: одно изменение ломает три других места, а понять «где же правило про отрицательный остаток» невозможно.

Решение DDD

DDD говорит: выдели «предметную область» (Domain) — сердце с чистыми бизнес-правилами — и не пускай туда технические детали. База данных, HTTP, кэш — это «périphérie», они снаружи и могут меняться, не задевая ядро.

Аналогия с рестораном. Domain — это рецепты и стандарты блюд (что считается правильным борщом). Infrastructure — это плита, холодильник, поставщик продуктов (их можно заменить, рецепт не изменится). UI — это официант, принимающий заказ. DDD требует: рецепт не должен зависеть от того, какой марки у вас холодильник.
Ключевой принцип: зависимости направлены внутрьВнешние слои (UI, Infrastructure) знают о ядре (Domain) и зависят от него. Но ядро не знает о них ничего. Product не подозревает, что существует PostgreSQL или HTTP. Благодаря этому ядро легко тестировать и оно не ломается при смене технологий.

5. Главные понятия DDD на примере этого проекта

5.1. Ограниченный контекст (Bounded Context)

Большую систему делят на самостоятельные части — контексты. У нас один контекст — Catalog (каталог). В большом магазине были бы ещё «Корзина», «Оплата», «Доставка». Каждый контекст — свой мир со своими правилами и своим словарём. Это видно по пути src/Catalog/… — всё лежит внутри одного контекста.

5.2. Сущность и Агрегат (Entity / Aggregate): Product

Сущность — объект, у которого есть уникальный идентификатор и своя «жизнь» во времени (его можно менять, но это всё тот же объект). Наш Product — это сущность и одновременно агрегат: главный объект, который отвечает за свою целостность. У товара есть артикул, название, описание, цена, остаток, даты создания/обновления и необязательная ссылка на Category.

Важно: правила живут внутри объекта. Нельзя положить отрицательный остаток — объект сам это запретит. Изменение категории тоже проходит через метод домена, а не через прямую запись поля из контроллера:

public function updateStock(int $newStock): void
{
    if ($newStock < 0) throw new \DomainException('Stock cannot be negative');
    $this->stock = $newStock;
    $this->updatedAt = new \DateTimeImmutable();
    $this->recordUpdated(); // запомнить, что товар изменился
}
public function assignCategory(?Category $category): void
{
    $this->category = $category;       // null означает «без категории»
    $this->updatedAt = new \DateTimeImmutable();
    $this->recordUpdated();
}
Почему это хорошоНевозможно «случайно» создать неправильный товар из другого места программы. Правило в одном месте — в самом объекте. Это и есть «защита инвариантов».

5.3. Сущность Category: раздел каталога

Category — отдельная сущность каталога. У неё свой UUID, название, slug, описание и даты. Товар может ссылаться на категорию, но категория живёт самостоятельно: её можно создать до товара, а потом назначить разным товарам.

class Category
{
    public const SLUG_PATTERN = '/^[a-z0-9]+(?:-[a-z0-9]+)*$/';

    private string $id;
    private string $name;
    private string $slug;

    private static function normalizeSlug(string $slug): string
    {
        $slug = strtolower(trim($slug));
        if (!preg_match(self::SLUG_PATTERN, $slug)) {
            throw new \InvalidArgumentException('Invalid category slug');
        }
        return $slug;
    }
}
Почему категория не Value ObjectКатегория имеет собственную идентичность и жизненный цикл: её можно создать, переименовать, найти по id или slug, а товары только ссылаются на неё. Поэтому это сущность, а не объект-значение.

5.4. Объект-значение (Value Object): Sku, Price, ProductId

Объект-значение — это маленький неизменяемый объект, который описывается только своим значением и не имеет собственного «id». Деньги, артикул, идентификатор — отличные кандидаты.

Зачем заворачивать цену в класс Price, а не хранить просто число? Чтобы правила и смысл были рядом со значением:

final class Price
{
    public function __construct(private int $amount, private string $currency = 'KZT')
    {
        if ($amount < 0) throw new \InvalidArgumentException('Price cannot be negative');
    }
    // два объекта-значения равны, если равны их значения
    public function equals(self $other): bool
    {
        return $this->amount === $other->amount && $this->currency === $other->currency;
    }
}
Аналогия. Купюра в 100 тенге равноценна любой другой купюре в 100 тенге — нам не важен её серийный номер. Это «объект-значение». А вот сотрудник Иван — это «сущность»: даже два полных тёзки — разные люди.

Артикул Sku хранит правило формата прямо в себе, в виде константы (её же переиспользует слой UI для проверки ввода — правило не дублируется):

final class Sku
{
    public const PATTERN = '/^[A-Z0-9]{8,20}$/'; // 8–20 заглавных букв и цифр
}

5.5. Доменные события (Domain Events): ProductCreatedEvent, ProductUpdatedEvent

Доменное событие — это факт «что-то важное произошло в бизнесе»: товар создан, товар изменён. Агрегат сам «записывает» такие факты, когда меняется. Позже их можно разослать другим частям системы (поиск, аналитика, уведомления) — и им не нужно для этого лезть внутрь нашего кода.

// внутри Product: накапливаем события, потом их «забирают»
public function pullDomainEvents(): array
{
    $events = $this->domainEvents;
    $this->domainEvents = []; // очищаем, чтобы не разослать дважды
    return $events;
}
Тонкость DDD, которую мы исправилиРаньше событие собиралось в обработчике (снаружи объекта). Это неправильно: «знание о том, что товар создан» — принадлежит самому товару. Теперь Product сам записывает событие в конструкторе, а обработчик лишь забирает и рассылает.

5.6. Репозиторий (Repository) и «порт»

Репозиторий — это «хранилище объектов»: умеет сохранить товар/категорию и найти их по id. Хитрость DDD: ядро объявляет только интерфейс (что должно уметь хранилище), а как именно — пишет Infrastructure.

// Domain: контракт (порт). Никакого SQL — только намерение.
interface ProductRepositoryInterface
{
    public function save(Product $product): void;
    public function findById(ProductId $id): ?Product;
    public function findByCriteria(array $filters, int $limit, int $offset): array;
}

interface CategoryRepositoryInterface
{
    public function save(Category $category): void;
    public function findById(string $id): ?Category;
    public function findBySlug(string $slug): ?Category;
}
// Infrastructure: реализация через Doctrine (вот тут уже база)
class ProductRepositoryDoctrine implements ProductRepositoryInterface
{
    public function save(Product $product): void
    {
        $this->em->persist($product);
        $this->em->flush(); // здесь Doctrine пишет SQL INSERT
    }
}
Зачем такая «двойственность»Это называется «инверсия зависимостей». Завтра вы смените PostgreSQL на MongoDB — поменяете только класс в Infrastructure, а всё ядро и сценарии останутся нетронутыми, потому что они зависят от интерфейса, а не от базы.

6. CQRS: команды и запросы

CQRS (Command Query Responsibility Segregation) — «разделение ответственности команд и запросов». Идея в одном предложении: операции, которые меняют данные, и операции, которые только читают данные, — это разные миры, и обрабатывать их надо по-разному.

Команда (Command)Запрос (Query)
ЦельИзменить состояние («создай товар», «создай категорию», «назначь категорию товару»)Прочитать состояние («дай товар», «дай список товаров», «дай категории»)
ВозвращаетНичего, подтверждение или read-модель изменённого объектаДанные для показа
В проектеCreateProductCommand, CreateCategoryCommand, AssignProductCategoryCommandGetProductQuery, ListProductsQuery, GetCategoryQuery, ListCategoriesQuery
Идёт через шинуcommand.busquery.bus

Команда — это просто «конверт с данными» о намерении. Сама она ничего не делает; её исполняет обработчик (handler):

class CreateProductCommand // просто данные намерения
{
    public function __construct(
        public readonly Sku $sku,
        public readonly string $name,
        public readonly Price $price,
        public readonly int $stock,
        public readonly ?string $description,
        public readonly ?string $categoryId = null
    ) {}
}

Для изменения категории у уже существующего товара используется отдельная команда. Это важно: создание товара и назначение категории — разные сценарии, и у каждого есть свой обработчик.

final class AssignProductCategoryCommand
{
    public function __construct(
        public readonly ProductId $productId,
        public readonly ?string $categoryId
    ) {}
}
Зачем разделятьЧтение часто можно ускорить (кэш, упрощённые модели данных), а запись — требует строгих проверок. Разделив их, мы оптимизируем каждую сторону отдельно. В нашем проекте чтение использует кэш Redis и лёгкую модель ProductView, а запись идёт через полноценный агрегат Product с проверками.

7. Механизмы Symfony, которые всё связывают

7.1. Контейнер и внедрение зависимостей (Dependency Injection)

Обработчикам вроде CreateProductHandler, CreateCategoryHandler и AssignProductCategoryHandler нужны репозитории, кэш и шина событий. Они их не создают сами — они просто заявляют в конструкторе «мне нужно вот это», а Symfony-контейнер при запуске сам создаёт и «вкладывает» нужные объекты. Это и есть внедрение зависимостей.

Аналогия. Вы не строите автомобиль, чтобы доехать до работы, — вам его «подают». Классу подают всё, что ему нужно, готовым.

В файле config/services.yaml мы говорим контейнеру, какую реализацию подставлять под интерфейс:

# «Когда кому-то нужен интерфейс репозитория — дай Doctrine-реализацию»
App\Catalog\Domain\Repository\ProductRepositoryInterface:
    '@App\Catalog\Infrastructure\Doctrine\ProductRepositoryDoctrine'
App\Catalog\Domain\Repository\CategoryRepositoryInterface:
    '@App\Catalog\Infrastructure\Doctrine\CategoryRepositoryDoctrine'

App\Catalog\Application\Port\ProductCacheInterface:
    '@App\Catalog\Infrastructure\Cache\ProductCacheRedis'

7.2. Messenger и три шины

Шина (bus) — это «почтовая служба»: вы бросаете в неё сообщение, она находит нужный обработчик и доставляет. У нас три шины под три типа сообщений (config/packages/messenger.yaml):

ШинаДля чего
command.busКоманды на изменение: создать товар, создать категорию, назначить категорию товару
query.busЗапросы на чтение: получить товар, список товаров, категорию, список категорий
event.busДоменные события (товар создан/изменён) — рассылаются асинхронно

7.3. Doctrine ORM: объекты ↔ таблицы

ORM (Object-Relational Mapping) избавляет от ручного написания SQL. Вы работаете с объектами Product и Category, а Doctrine знает, в какие таблицы и колонки их положить. «Карта» соответствия задана XML-файлами в Infrastructure/Doctrine/Mapping.

Объекты-значения Sku и Price хранятся не в отдельных таблицах, а «встраиваются» в строку товара (это называется embeddable): цена превращается в две колонки — price_amount и price_currency.

Связь товара с категорией хранится как nullable many-to-one: в таблице catalog_products есть колонка category_id, которая ссылается на catalog_categories.id. Nullable означает, что товар может существовать без категории. В XML-маппинге это выглядит так:

<many-to-one field="category" target-entity="App\Catalog\Domain\Entity\Category">
    <join-column name="category_id" referenced-column-name="id"
                 nullable="true" on-delete="SET NULL"/>
</many-to-one>

А чтобы наш ProductId (объект) корректно превращался в строку UUID для базы и обратно, написан кастомный тип ProductIdType:

// объект ProductId → строка для базы
public function convertToDatabaseValue($value, $platform): ?string
{
    return $value instanceof ProductId ? $value->toString() : null;
}

7.4. Кэш Redis за «портом»

Кэш — быстрая память для часто запрашиваемых данных, чтобы не дёргать базу каждый раз. Redis — популярное хранилище для этого. Чтобы слой Application не зависел от Redis напрямую, мы спрятали кэш за интерфейс-порт ProductCacheInterface; реализация ProductCacheRedis живёт в Infrastructure. Кэш хранит лёгкую read-модель ProductView на 5 минут.

Когда категория существующего товара меняется через AssignProductCategoryHandler, кэш конкретного товара инвалидируется: следующий GET /api/catalog/products/{id} уже вернёт актуальную категорию.

7.5. Контроллер и проверка входных данных

Контроллер — «дверь» приложения. Symfony сам превращает входной JSON в объект запроса и проверяет его (атрибут #[MapRequestPayload]). Если данные кривые — клиент получит ошибку ещё до того, как они дойдут до бизнес-логики.

#[Route('/api/catalog/products', methods: ['POST'])]
public function create(#[MapRequestPayload] CreateProductRequest $request): JsonResponse

Для разных действий используются разные DTO: CreateProductRequest проверяет товар, CreateCategoryRequest проверяет название и slug категории, а AssignProductCategoryRequest проверяет, что categoryId либо UUID, либо null.

8. Сквозной сценарий №1: создание товара (POST)

Проследим, что происходит, когда приходит POST /api/catalog/products с JSON товара. В запросе можно передать categoryId, но это поле необязательное: товар может быть создан без категории.

1UI. Symfony по адресу находит метод ProductController::create. JSON автоматически превращается в CreateProductRequest и проверяется (артикул, длина имени, цена > 0, остаток ≥ 0, categoryId — UUID или null).
2UI → Application. Контроллер собирает из проверенных данных CreateProductCommand и бросает её в command.bus.
3Application. Если categoryId передан, CreateProductHandler ищет категорию через CategoryRepositoryInterface. Если категории нет — возвращается ошибка 400 Bad Request.
4Domain. Обработчик создаёт новый агрегат new Product(..., $category). Товар проверяет свои правила и записывает событие «товар создан».
5Application → Infrastructure. Обработчик вызывает repository->save($product). Doctrine-репозиторий пишет строку в PostgreSQL.
6События. Обработчик забирает у товара события (pullDomainEvents()) и рассылает их в event.bus.
7Ответ. Контроллер возвращает JSON {"status":"Product created"} и HTTP-код 201 Created.
// Фрагмент CreateProductHandler
$category = null;
if ($command->categoryId !== null) {
    $category = $this->categoryRepository->findById($command->categoryId);
    if (!$category) {
        throw new \DomainException('Category not found');
    }
}

$product = new Product($command->sku, $command->name, $command->price,
                      $command->stock, $command->description, $category);
$this->repository->save($product);

9. Сквозной сценарий №2: чтение и список товаров (GET)

В Catalog есть два сценария чтения товаров: получить один товар по id и получить список товаров. Оба возвращают read-модель ProductView, где категория вложена как объект CategoryView или равна null.

9.1. Один товар: GET /api/catalog/products/{id}

1UI. ProductController::get превращает строку {id} в объект ProductId. Если id невалидный — сразу ответ 400 Bad Request.
2UI → Application. Контроллер бросает GetProductQuery в query.bus; шина возвращает результат обработчика.
3Application + кэш. GetProductHandler спрашивает кэш: «есть товар с таким id?». Если есть — отдаёт мгновенно. Если нет — идёт в репозиторий.
4Ответ. Если товар найден — ProductView превращается в JSON (код 200 OK). Если нет — 404 Not Found.
public function __invoke(GetProductQuery $query): ?ProductView
{
    return $this->cache->get($query->id, function () use ($query): ?ProductView {
        $product = $this->repository->findById($query->id);   // промах кэша → база
        return $product ? ProductView::fromEntity($product) : null;
    });
}

9.2. Список товаров: GET /api/catalog/products

Список товаров не использует кэш одного товара, потому что это другой read-сценарий: у него есть фильтры, лимит и смещение. Сейчас поддерживаются:

  • q — поиск по части названия;
  • categoryId — оставить только товары выбранной категории;
  • limit и offset — простая пагинация.
if ($query->categoryId !== null && $query->categoryId !== '') {
    $filters['categoryId'] = $query->categoryId;
}

return array_map(
    static fn ($product): ProductView => ProductView::fromEntity($product),
    $this->repository->findByCriteria($filters, $query->limit, $query->offset)
);
Почему отдаём ProductView, а не ProductТяжёлый агрегат с правилами незачем (и небезопасно) складывать в кэш и слать наружу. Лёгкая «витрина» ProductView содержит только то, что нужно показать, и легко превращается в JSON.

10. Сквозной сценарий №3: категории и привязка товара к категории

Категории добавлены в тот же bounded context Catalog, потому что это часть языка каталога: товар может быть «без категории» или находиться в конкретном разделе.

10.1. Создание категории

1UI. CategoryController::create принимает JSON и валидирует name, slug, description.
2Application. CreateCategoryHandler проверяет уникальность slug через CategoryRepositoryInterface::findBySlug().
3Domain. Создаётся Category; сама сущность нормализует имя и slug.
4Ответ. Контроллер возвращает созданную категорию как CategoryView и HTTP-код 201 Created.

10.2. Привязка категории к уже существующему товару

Для админ-панели нужен сценарий «товар уже создан, теперь выбери ему категорию». Это не должно имитироваться на frontend: выбор должен сохраняться в backend. Поэтому добавлен endpoint PATCH /api/catalog/products/{id}/category.

1UI. ProductController::assignCategory проверяет id товара из URL и payload {"categoryId": "uuid"} или {"categoryId": null}.
2Application. AssignProductCategoryHandler находит товар. Если товара нет — 404 Not Found.
3Application. Если categoryId не null, обработчик находит категорию. Если категории нет — 404 Not Found.
4Domain. Вызывается $product->assignCategory($category). Товар обновляет updatedAt и записывает ProductUpdatedEvent.
5Infrastructure. Обработчик сохраняет товар, инвалидирует Redis-кэш этого товара и рассылает доменные события.
6Ответ. Возвращается обновлённый ProductView вместе с актуальной категорией.
public function __invoke(AssignProductCategoryCommand $command): ProductView
{
    $product = $this->productRepository->findById($command->productId);
    if (!$product) throw new \DomainException('Product not found');

    $category = null;
    if ($command->categoryId !== null && $command->categoryId !== '') {
        $category = $this->categoryRepository->findById($command->categoryId);
        if (!$category) throw new \DomainException('Category not found');
    }

    $product->assignCategory($category);
    $this->productRepository->save($product);
    $this->productCache->invalidate($product->getId());

    return ProductView::fromEntity($product);
}

13. Зачем всё это: выгоды подхода

Слоёв и абстракций больше, чем в «простом» коде — и это окупается, когда проект растёт:

  • Бизнес-правила в одном месте. Хотите узнать все правила про товар — открываете Product; правила категории — Category.
  • Легко тестировать. Ядро не зависит от базы и сети, поэтому правила проверяются быстрыми unit-тестами без запуска PostgreSQL/Redis. В проекте уже есть такие тесты (12 шт.) — они проверяют события агрегата и объекты-значения за доли секунды.
  • Технологии заменяемы. Сменить базу или кэш = поменять класс в Infrastructure. Ядро и сценарии не трогаются.
  • Чтение и запись масштабируются раздельно (благодаря CQRS): читаем — из кэша и быстрых моделей, пишем — со строгими проверками.
  • Понятный словарь. Код говорит на языке бизнеса (Product, Category, Sku, Price, «товар создан», «категория назначена»), а не на языке таблиц.
Честно о минусахDDD — это «инвестиция». Для крошечного скрипта он избыточен: больше файлов, больше понятий. Он оправдан там, где логика сложная и проект живёт долго (как раз случай растущего магазина — «highload»-приложения, судя по названию проекта).
